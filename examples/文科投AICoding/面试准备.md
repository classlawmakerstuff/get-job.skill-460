## 一、虚构岗位 JD（先把岗位摸清楚）

**岗位**：AI Product Prototyping Intern / AI 应用产品原型实习生
**虚构公司**：一家做 AI workflow、活动/社区效率工具、内部 AI assistant 的早期产品团队
**岗位定位**：不是纯算法岗，也不是纯前端岗，而是偏"产品理解 + AI coding + 可运行 demo"的应用型岗位。核心价值是把真实场景里的模糊痛点，快速拆成产品流程、数据结构、交互页面和可演示原型。

**Responsibilities**
- 和产品/设计/工程团队一起调研真实用户场景，把一线问题转化为 AI 应用需求、用户故事和 demo 范围
- 用 AI coding 工具快速搭 Web demo：基础前端页面、后端接口、轻量数据库、导出和报告生成
- 设计表单输入、数据记录、状态流转、导出统计等工作流
- 把线下的人工流程转化为可复用的数据字段，并设计低负担、可解释的交互
- 基于测试反馈迭代 prompt、页面流程、错误处理、边界场景和产品提示文案
- 能清楚说明 demo 背后的产品判断、技术取舍、用户风险和下一步迭代方向

**Requirements**
- 对 AI 应用产品有兴趣，理解 LLM / AI coding 适合做什么、不适合做什么
- 有从 0 到 1 做 demo 或课程/个人项目的经历，能展示真实可运行的原型
- 熟悉基础 Web 开发或愿意用 AI coding 工具完成前后端功能拆解
- 能从真实场景中提炼需求，不只停留在"做一个 AI 工具"的宽泛想法
- 具备数据意识，能设计字段、记录结构、导出格式或简单统计逻辑
- 表达清楚，能把专业议题翻译成普通用户也能理解的产品叙事
- 加分项：有活动运营、校园项目、教育、社区等真实场景观察经验

## 三、1 分钟自我介绍模板（两个版本）

**版本 A：偏产品 / AI 应用**
> "我比较适合这个岗位的点，是我不是单纯从技术出发做 demo，而是会先从真实场景里拆问题。之前我在校园活动里发现报名表散落各处、签到靠人工点名、活动后统计很累，就用 AI coding 做了一个集报名、签到、出勤导出于一体的轻量工具 EventEase。这个项目里我负责从用户痛点、数据模型、页面流程到前后端 demo 的完整拆解，用 React、Python backend 和 SQLite 做出了本地可运行版本。我更关心怎么让组织者几步就能完成、让参与者一眼看懂——所以做了二维码签到和一键导出，而不是堆功能。"

**版本 B：偏 AI coding / demo 落地**
> "我过去的项目主要围绕 AI 应用原型和 Web demo。最完整的一个是面向活动组织场景的报名签到工具，我把线下'发表格—点名—事后整理 Excel'的流程，做成了报名表单、二维码签到、一键出勤导出。这个过程中我用 AI coding 辅助拆解用户故事、页面结构、接口字段和签到逻辑，但核心产品判断是我自己做的：哪些步骤要自动化、哪些要保留人工、哪些字段才真的有助于后续统计。"

## 四、核心项目怎么讲：从"普通项目"讲成"岗位匹配项目"

推荐讲法：**问题 → 用户 → 产品方案 → 技术实现 → 风险边界 → 结果/反思**

1. **问题**：校园社团办活动时，报名表散落在各种群和文档里，签到靠人工点名，活动后还要手动整理出勤，很累也容易出错。
2. **用户**：社团/活动组织者，以及报名参加的同学。他们不是技术人员，工具必须简单。
3. **方案**：做一个报名表单 + 二维码签到 + 一键出勤导出的轻量工具，把几步流程压缩。
4. **技术**：前端用 React/Tailwind 做表单和签到页面，后端用 Python 接收记录，SQLite 保存 event/participant/签到时间，支持 CSV 导出。
5. **边界**：自动提醒只做"临近活动提示未签到"，不做骚扰式群发；数据只用于出勤统计。
6. **反思**：这个项目不是为了证明技术多厉害，而是证明能把一个真实的人工流程，用 AI coding 快速做成省事的工具。
